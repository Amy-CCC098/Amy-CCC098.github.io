wireshark的使用

工具简述

1、wireshark工具的使用

Wireshark是一款抓包工具，能获取HTTP，也能获取HTTPS，但是不能解密HTTPS。

打开wireshark，选择要监听的端口，然后开始抓包。

抓包界面如下图所示，包的详细信息里对应OSI七层模型：

在嗅探包的时候会有很多各种各样的包，所以我们可以通过设置过滤来查看我们想要的包。

1.1地址过滤

  • 指定源ip：ip.src == IP
  • 指定目的ip：ip.dst == IP
  • 与某ip的交互：ip.addr == IP
  • 指定的源ip或指定的目的ip: ip.src == IP or ip.dst == IP
  • 指定的源ip并且指定的目的ip: ip.src == IP and ip.dst == IP

1.2端口过滤

  • 可以指定协议是tcp还是udp，然后用srcport，dstport，port来确定要指定的是源端口还是目的端口。

1.3协议过滤

  • 协议有tcp，udp，arp，icmp，http，smtp，ftp，dns，ssl等，要指定某个协议可以在过滤栏输入协议名称。不要某个协议过滤栏输入!协议名称。

1.4 get/post过滤

  • http.request.method == GET
  • http.request.method == POST

2、Wireshark抓包原理

​ wireshark常见抓到包的类型有:ARP协议、ICMP协议、TCP协议、UDP协议、DNS协议、HTTP协议，在本机环境下它抓包抓取的是我们电脑的网卡进出的流量。Wireshark监听一个端口，比如8888，在测试机上连接同一个局域网，配置网络代理，指向该主机的8888端口，这样一来测试机的所有网络通信都会被转发到PC的8888端口，进而被wireshark捕获，然后就可以对数据包进行分析。

实际操作

使用wireshark进行一次网站访问

  1. 命令行输入nslookup，再输入网址cclab.ujs.edu.cn，发现其ip为218.3.94.83，所以在抓包过程中可以用ip.addr==218.3.94.83进行过滤

  2. 访问网站cclab.ujs.edu.cn

  3. 用wireshark对数据包进行查看

对整个网站访问过程抓取的数据包进行分析

a.TCP连接和断开三次握手

​ 下图是wireshark抓取到的访问cclab.ujs.edu.cn时三次握手的过程

  • 第一次握手，客户端发送一个TCP，标志位SYN为１，序号seq=0,36110->80代表客户端请求连接。

    本机地址192.168.0.103向目的地址218.3.94.83发送请求。

  • 第二次握手是服务器向客户端返回一个数据包，80->36110，序列号为０，确认序号为１。

  • 第三次握手是在第五行，第三第四行是进行了一个重发，客户端收到服务器发来的包之后检查确认序号是否为正确，以及标志位是否为１。如果正确，客户端会向服务器端发送一个数据包，36110->80，发送的数据包中确认序列ack=1，发送序列seq=0。至此，一次三次握手，TCP连接建立完成。

b.七层数据分析

​ 在封包详细信息处，从上至下分别为物理层、数据链路层、IP网络层、TCP传输层以及最后面的HTTP

  • 物理层中可以看到包到达的事件，与之前捕获的数据帧查理0.001174330s，和之前显示的数据帧差了0.001174330s，包的长度为393bytes。

  • 数据链路层中可以看到本地的mac地址和目标的mac地址,其中type:IPv4是发送的数据包类型。

  • 网络层中可以看到头部的长度为20bytes，以及源地址和目标地址的ip。

  • 传输层中可以看到源端口和目标端口，报文长度，序列号，确认序号等，这里用的是TCP协议传输。

  • http报文中可以看到访问该网页用的浏览器，语言，编码，访问网站的域名等。

c.分析SSL握手情况

Https实现安全通信就是再http和TCP之间加入SSL协议层，SSL协议中客户机和服务器会进行四次握手。

  • 第一次握手客户端发送给服务器端。客户端发送Client Hello的消息，其报文内容有：客户端TLS版本号（TLS 1.2）、长度、生成的随机数、加密套件列表（Cipher Suites）等。

  • 第二次握手服务器端发给客户端。服务端收到客户端的Client Hello的消息，回复Server Hello。其报文内容有：服务器确认客户端TLS版本号（TLS 1.2）、长度、服务器生成的随机数、服务器从客户机加密套件列表中选出一个最合适的加密组合等，这里选择的是TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 (0xc02f)，表示用128位以GCM作为分组模式的AES进行加密，用RSA进行身份认证，ECDHE作为密码交换算法，SHA256作为摘要算法。

​ 之后，服务器会发送一个认证信息，里面包括CA给服务器颁发的公钥、CA机构的签名等。以及发送一个服务器密钥交换消息以及通知客户端第二次握手中服务器的所有消息都已发送完毕的消息。

  • 第三次握手时，客户端发给服务器端。客户端在收到服务器返回的 Server Hello 和 Certificate 消息后，使用公钥对证书进行解密，得到数据原文及摘要值1，然后客户端使用与CA相同的摘要算法对数据原文进行计算得到摘要值2，比较摘要值1与摘要值2，若完全相同则说明证书合法且未被其他人篡改，从而拿到服务器的RSA公钥，并发送客户端密钥交换消息。从抓到的数据包中我们还可以看到diffie-hellman密钥交换协议的公钥为:f479d1a88d6d73018dba2b2b69ae2a8d2c71a345284b08ea57df544aae843a6e。

    然后发送信息Change Cipher Spec，将对称密钥发送给服务器，并通知服务器开始使用对称加密的方式进行通信。

    最后，客户端发送Encrypted Handshake Message 消息，将之前发送的所有数据做成摘要，使用对称密钥加密，供服务器验证之前握手过程中的数据是否被其他人篡改。

  • 第四次握手是服务器端发给客户端。服务器端先向客户端发送一个New Session Ticket的消息。然后，再用RSA私钥对Client Key Exchange消息解密，得到客户端的随机数，并计算对称密钥，将计算后的结果通过Change Cipher Spec消息返回给客户端。

    最后，服务器再发送一个Encrypted Handshake Message消息，将握手过程中的对称密钥加密后发给客户端。TLS四次握手完毕。
